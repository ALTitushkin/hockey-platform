# Требования и триггеры масштабирования

> Явные пороги: когда текущего стека (GitHub Pages + один общий VPS + данные в JSON/git) достаточно, а когда пора добавлять ресурс. Держим под контролем, чтобы не масштабироваться «на всякий случай» и не проспать реальный предел.

## Текущий стек и его ёмкость
- **Сайт:** статика на GitHub Pages (CDN Fastly). Чтения — практически неограниченно, бесплатно. Предел — только если понадобится серверная логика (auth, динамический API, SSR).
- **Бот:** один процесс `python-telegram-bot` (polling) на общем VPS, читает JSON в память. Данные крошечные (<1 МБ). Стейтлесс.
- **Данные:** курируемые JSON в git (terms/history/calendar/clusters). Нет БД, нет пользовательских данных, нет конкурентной записи.
- **VPS:** общий с x-ui (VPN) и nginx (опросник). ⚠️ **Свободно ~230 МБ RAM — это узкое место №1.**
- **CI:** GitHub Actions, бесплатный тариф. При нашем темпе — с запасом.

## Таблица решений: хватает / пора добавлять

| Область | Пока хватает GitHub+VPS | 🚨 Триггер добавить ресурс | Что добавляем |
|---|---|---|---|
| **RAM VPS** | Свободно > ~120 МБ | Свободно < ~100 МБ или OOM в `journalctl` | Апгрейд VPS / вынести бота на отдельный инстанс |
| **Трафик бота** | Polling справляется, отклик быстрый | Задержки ответов, рост нагрузки, нужен масштаб | Перейти на webhook; при большом объёме — dedicated хост |
| **Данные** | Курируемый JSON, читается в память, нет юзер-данных | Появились данные ПОЛЬЗОВАТЕЛЯ / конкурентная запись / > ~10k записей / нужны запросы | SQLite → при росте Postgres |
| **Сайт** | Только статика (словарь/история/бот-ссылки) | Нужны auth, серверный API, персонализация, SSR | Отдельный app-сервер (VPS/контейнер), не Pages |
| **Фоновые задачи** | 1–2 задачи (напр. «саммари дня») | Много задач/расписаний, ретраи, очереди | systemd-timer → при росте job-scheduler/очередь |
| **Медиа/хранилище** | Ссылки на внешние видео (YouTube) | Свой хостинг картинок/видео, большой объём | Object storage (S3-совместимое) + CDN |
| **CI-минуты** | Редкие пуши, лёгкие джобы | Упёрлись в лимит Actions | Платный тариф / self-hosted runner |
| **Аптайм/критичность** | Пет-проект, редкие падения терпимы | Бот стал важен, простой недопустим | Мониторинг + авто-рестарт + резерв |

## Правила
- **Не масштабируемся заранее.** Добавляем ресурс только по сработавшему триггеру, а не «на будущее».
- **RAM VPS — под контролем.** Раз в спринт (или при новой фиче бота) проверять свободную память: `free -m` на VPS. Заносить в разбор логов.
- **Первый шаг всегда дешёвый.** SQLite до Postgres; systemd-timer до очередей; webhook до отдельного хоста. Прыгать через ступень — только по явной необходимости.
- **Пользовательские данные меняют всё.** Как только храним что-то про конкретного юзера — сразу нужна БД, бэкапы и приватность (сейчас этого нет и не нужно).

## Ближайший релевантный кейс
**Scheduled task «саммари дня в канал» (кандидат S8):** укладывается в текущий стек — `systemd-timer` на VPS + существующий `calendar.json`. Новых ресурсов НЕ требует. Следить только за RAM (см. правило выше).
