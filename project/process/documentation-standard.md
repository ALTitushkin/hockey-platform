# Стандарт ведения документации

> Как мы держим документацию — на этом проекте и на всех последующих. Основано на практиках Diátaxis / GitBook / Read the Docs.
> Для нового проекта: скопировать этот файл + структуру `project/` как шаблон.

## Принципы

1. **Продукт ≠ документация.** Исходники сайта/приложения и проектная дока лежат в РАЗНЫХ папках. У нас: сайт → `docs/` (так требует GitHub Pages), дока → `project/`. Не смешивать.
2. **Единая точка входа.** `project/README.md` — оглавление со ссылками на все разделы. Человек начинает отсюда, не роется в файлах.
3. **Группировка по назначению, не свалкой:**
   - `planning/` — ТЗ спринтов (`sprint_N.md`) + `results/` (итоги/ретро `sprint_N_results.md`).
   - `design/` — концепт, дизайн-брифы, спеки визуала.
   - `content/` — правила наполнения, рубрикаторы, черновики публичных текстов.
   - `process/` — как работаем: операционка (грабли/инфра), роли/процесс, архитектурные решения, этот стандарт.
   - `handoff/` — готовые ТЗ-передачи PM → исполнитель по трекам (шаблоны).
4. **Консистентные имена.** `kebab-case`, самоописательные, без транслита-загадок (плохо: `anons_kanal`; хорошо: `channel-announcement`). Английские слоги для файлов, русский — внутри.
5. **Два слоя аудитории:**
   - *Витрина (для человека)* — Obsidian-дашборд: что за проект, статус, по каждому спринту «сделано / не сделано / планируем». Ёмко и атомарно.
   - *Рабочий слой (для Claude/дева)* — репо: детали спринтов, операционка, брифы. Обновляется синхронно с витриной.
6. **Каждый спринт = ТЗ + итоги.** ТЗ до старта (`sprint_N.md`, scope+DoD), итоги при закрытии (`sprint_N_results.md`, что вышло + уроки + перенос).
7. **Diátaxis для пользовательских доков.** Если появятся доки для конечного пользователя продукта — делить на tutorial / how-to / reference / explanation. Внутренние проектные доки делим по назначению (см. п.3).
8. **Статусы без хвостов.** При закрытии спринта прогонять блок на остаточные `[ ]`; репо и дашборд держать синхронными.

## Карта «что где лежит» (этот проект)
- Сайт (GitHub Pages): `docs/`
- Код: `bot/`, `tools/` · Данные: `data/` (+ зеркало `docs/data/`)
- Контекст для Claude: `CLAUDE.md` (читается первым) · Версии: `CHANGELOG.md`
- Проектная дока: `project/` (вход — `project/README.md`)
- Живой статус: Obsidian-дашборд (у владельца)

## Чек-лист при заведении нового проекта
- [ ] `project/` со структурой planning/design/content/process/handoff + `README.md`.
- [ ] Сайт/приложение — в отдельной папке, не смешан с докой.
- [ ] `CLAUDE.md` с деревом структуры и ссылкой на `project/README.md`.
- [ ] Дашборд-витрина (Obsidian или аналог) синхронно с репо.
