- **Секреты — чисто.** `.env` в `.gitignore`, в индексе секретов нет, деплой-ключ отдельный (не личный), VPS-креды в GitHub Secrets. ✅

- **systemd закалён:** `Restart=on-failure`, `RestartSec=10`, `NoNewPrivileges`, `ProtectSystem=strict`, `ProtectHome`, отдельный юзер `hockey`, `After=network-online`. ✅
- **Деплой безопасен:** `set -e`, `git pull --ff-only`, установка зависимостей, рестарт. ✅
- **Данные = бэкап:** база в git (terms/history/calendar.json), бот стейтлесс → потеря VPS не теряет данные. ✅

## Гэпы (по приоритету)

### 🔴 P1 — деплой без тестов (главный риск)
`deploy.yml` сразу рестартит бота, НЕ прогоняя `test_search.py`/`test_calendar.py`. Был реальный кейс обрезанного `bot.py` — прод падал.
**Фикс:** отдельный job `test` (pytest) → `deploy` зависит от него (`needs: test`). Деплой только если тесты зелёные. + прогон `validate_data.py` на изменения `data/**`.

### 🔴 P1 — слабый healthcheck после деплоя
Проверяется только `systemctl is-active` — а известная грабля в том, что процесс стартует и тихо выходит (is-active ещё «activating/active»).
**Фикс:** после рестарта грепать `journalctl -u hockey-bot --since "10 sec ago"` на строку «Бот запущен …»; если нет — падать с ошибкой (workflow краснеет) и НЕ считать деплой успешным.

### 🟡 P2 — нет уведомления о падении
Если бот умер и systemd не поднял (битый деплой/OOM) — никто не узнает.
**Фикс:** лёгкий healthcheck-крон на VPS или внешний аптайм-монитор → алерт в Telegram админу. Дёшево, снимает «узнаю, когда пользователь пожалуется».

### 🔴 P1 — Бот живёт на личном сервере (решено переезжать, 19.07.2026)
Хоккей-бот — публичное приложение, но крутится на личной машине вместе с WireGuard-VPN и Obsidian GitSync, на 1 ГБ памяти. Нарушает принцип «личное ≠ публичное»: компрометация бота, OOM или DDoS задевают личную инфраструктуру. Подсвечено инцидентом с утечкой токена 19.07.
**Фикс:** переезд на app-VPS (4 ГБ, Docker, изоляция) — тем же путём, которым 11.07 уехал PrizeDrawing. Черновой скоуп — в Obsidian-дашборде проекта. Планируется отдельным треком, **не в S8** (S8 занят конвейером календаря).

### 🟡 P2 — VPS-конфиг не забэкаплен / нет runbook
Данные в git, а конфиг VPS (systemd, nginx, x-ui) — нет. При потере VPS — восстановление по памяти.
**Фикс:** runbook «поднять бота с нуля» (уже частично в `ops_notes.md`) + держать конфиги в репо (`deploy/` уже есть для юнита).

### 🟢 P3 — воспроизводимость зависимостей
`requirements.txt` пинит только PTB (`>=21,<22`). Одна зависимость — ок, но лок-файл (`pip freeze`) даёт полную повторяемость.
**Фикс (по желанию):** зафиксировать транзитивные версии, когда зависимостей станет больше.

### 🟢 P3 — CI для сайта
Нет валидации HTML/JSON-LD/битых ссылок для `docs/`. При росте истории — полезно.
**Фикс (по желанию):** линк-чекер + JSON-LD-валидатор в CI на `docs/**`.

## План рефакторинга (предлагаемый порядок)
1. P1: тесты перед деплоем + `validate_data.py` в CI (правка `deploy.yml`).
2. P1: healthcheck по `journalctl` вместо `is-active`.
3. P2: алерт о падении (крон + Telegram).
4. P2: runbook восстановления.
5. P3 — по мере роста.
