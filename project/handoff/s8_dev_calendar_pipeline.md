# Бриф деву · S8 · Трек A — конвейер календаря

> PM → Dev (Cowork) · 18.07.2026 · Sprint 8, трек A (P0)

---

## Зачем

Исследователь начинает сдавать календарь батчами по 2–3 месяца (`s8_research_calendar.md`). Сейчас данные попадают в бота **вручную**: файл лежит в `hockey_calendar/hockey_calendar.json` в корне воркспейса, оттуда копируется в `data/calendar.json`. На S7 это сделали руками один раз и записали в уроки (№3: «источник данных вне репо»).

Если оставить как есть — каждый батч будет требовать ручной операции, и рано или поздно в проде окажется несинхронизированный или битый файл.

**Задача трека:** сданный батч доезжает до бота одной командой, а битый батч ломает CI до деплоя, а не в проде.

---

## Что сделать

### 1. Авто-синк источника

Скрипт в `tools/` (имя на твоё усмотрение, напр. `sync_calendar.py`):
`hockey_calendar/hockey_calendar.json` → `data/calendar.json`.

- Путь к источнику — параметром или через конфиг, не хардкодом абсолютного пути (файл живёт вне репо).
- Если источник недоступен — внятная ошибка, не тихий пропуск.
- Идемпотентность: повторный запуск без изменений не создаёт diff.
- **Паритет с `docs/data/` не нужен** — календарь читает только бот, сайт его не использует (см. `ops_notes.md`).

### 2. `tools/validate_calendar.py`

Гейтит деплой наравне с `validate_data.py`. Проверки:

**Формат**
- ключ даты — строго `MM-DD`, существующая дата (учесть `02-29`);
- обязательные поля события: `year`, `text`, `cat`, `reg`, `conf`;
- `cat` ∈ {Рождение, Гибель, Дебют, Рекорд, Матч, Турнир, Переход, Возвращение, Достижение, Организация};
- `reg` ∈ {RU, NHL, INT, WORLD};
- `conf` ∈ {high, med};
- `video` — если поле есть, это синтаксически валидный URL (живость ссылки не проверяем, это сеть в CI);
- `year` — правдоподобный диапазон (скажем, 1875…текущий год), чтобы ловить опечатки вроде `19555`.

**Целостность**
- нет дублей событий внутри одной даты (совпадение `year` + нормализованный `text`);
- `text` не пустой.

**Отчёт покрытия** (печатать всегда, не только при ошибке):
```
календарь: 174/366 дней, 210 событий
пустых: 192 · макс. дыра: 9 дн. (08-18 … 08-26)
по месяцам: 01:12/31 02:11/29 ...
```

Ошибки формата — **exit 1** (валит CI). Низкое покрытие — **не ошибка**, просто отчёт: календарь наполняется постепенно, это нормальное состояние.

### 3. Счётчик покрытия в логе бота

При старте бот пишет строку вида `календарь: N/366, макс. дыра M дн.` — по образцу счётчика терминов `62 (+3 на выверке)` из S7. Чтобы прогресс был виден в `journalctl` без запуска скриптов.

## Границы

- **Постинг в канал в этот спринт НЕ делаем.** Scheduled task уезжает в S9, включается по гейту (покрытие ≥90%, нет дыр >2 дней). Не начинай его «заодно».
- Логику `calendar_feature.py` не трогаем, кроме пункта из P2 ниже.
- Формат данных не меняем — исследователь уже пишет в него.

---

## Грабли (из опыта проекта)

- Модуль называется `calendar_feature.py`, а не `calendar.py` — не затеняем stdlib. Новые файлы называй так же осторожно.
- Даты — МСК. Календарь по ключу `MM-DD`, без года.
- `02-29` — валидный ключ, обрабатывать без падения в невисокосный год.
- Ruff: линт чистый на `E,F @120`, ruff блокирует деплой. Новый код должен проходить.
