## 2. Проблема и цель

**Сейчас:** сайт = плоский словарь, история — ссылка в подвале, теги уровня сделаны «от балды».
**Цель:** спроектировать каркас, где платформа ощущается образовательным ядром «Культура хоккея», а не справочником, — и который дорастает до обучения/сообщества без редизайна.

**Формула платформы (ось всего):** вопрос → краткое выверенное саммари → якорь(и) в конкретное место каноничного материала. От «что такое форчек» до «как поставить зонную оборону» — один механизм, разная глубина. **Без LLM** — движок это структура + якоря.

---

## 3. Жёсткие ограничения (не обсуждается)

- **Статика:** GitHub Pages, без бэкенда/БД. Один контент = страница, навигация по якорям.
- **Mobile-first:** основной трафик с телефона из Telegram.
- **Данные отдельно от UI:** разметка — повторяющиеся блоки с классами, контент подставляется из JSON. Дизайнер даёт примеры с реальным содержимым, не финальную базу.
- **Кириллица обязательна**, тёмная тема не нужна.
- **Ресурс:** автор один, ~2 ч/нед. Каркас должен быть реализуем поэтапно, без «заработает только когда достроим всё».

## 4. Что спроектировать

1. **Информационная архитектура / карта сайта.** Ответ-вход на главной + **равноправные** разделы: Словарь · История · (пустой слот) Обучение · (пустой слот) Сообщество. История выходит из подвала. Слоты «Обучение/Сообщество» — спроектировать как заглушки-места, НЕ наполнять.
2. **Ответ-вход («спроси»).** Поле вопроса на главной → выдаёт краткое саммари + deep-link(и) на якорь в нужном материале (термин / глава / раздел). Спроектировать состояния: подсказки, найдено (термин), найдено (история), не найдено. Без живого чат-бота — это маршрутизатор по структуре.
3. **Словарь — две оси таксономии вместо мешанины.** Фильтры по **Уровню** (база → продвинутый, прогрессия) и по **Профилю** (общий, игрок, тренер, аналитик, скаут…) — независимые, комбинируются. Показать, как это уживается с текущими категориями (Статистика/Тактика/Правила/Позиции).
4. **Кластеры и связи терминов.** Из плоского списка — в смысловые группы с вложенностью (зонтик ⊂ большинство; кистевой/щелчок/подщёлкивание ⊂ броски) + блок «см. также». Карточка термина → должна вмещать связи и якорь «подробнее в истории/гайде».
5. **Визуализация как первоклассный контент.** Слот «схема и/или видео» в карточке термина и в тексте главы. Учесть реалии: SVG-схемы свои (стиль `diagrams/`), видео — встраиваемое чужое (YouTube + таймкод). Спроектировать оба слота.
6. **Раздел История** как равноправный: индекс глав (тизеры), страница главы с якорями на разделы (deep-link цель для ответ-входа), навигация между главами.
7. **Идентичность — открытый вопрос (см. §6).**

---

## 5. Чего НЕ делать

- Не наполнять слоты «Обучение» и «Сообщество» — только заложить места в IA.
- Не проектировать «север» (вузы/MBA, агрегатор проблем, адвокаси) — это другой класс продукта, далеко.
- Не писать прод-код и не трогать `terms.json`/бота — это дев-чат. Твой выход — концепт + примеры + промпт.
- Не закладывать живой LLM-ответ — только маршрутизация по якорям.

## 6. Имя и идентичность

**Имя — решено:** платформа называется **«Культура хоккея»** (вордмарк в шапке) + строка «при поддержке канала „Хоккейный Овертайм“» со ссылкой. Бот — «Учебник хоккея». Канал не переименовываем.

**Идентичность — открытый вопрос.** Стиль «программка матча» (кремовый фон, Oswald+PT Serif+PT Mono, ретро-рамки — см. `design_brief.md`) — это ДНК эпохи словаря. Вопрос: **тянет ли он на «учебник/платформу» или нужна эволюция?** Дай 2–3 направления настроения (напр.: «строгий ретро-учебник», «тёплый ламповый», «современный с ретро-акцентами») — выбирать проще, чем формулировать вслепую. Не ломай узнаваемость с каналом «Хоккейный Овертайм».

## 7. Ожидаемые результаты (отдать в PM-чат)

1. **IA / карта сайта** — разделы, иерархия, где слоты под рост.
2. **Логика ключевых экранов** (вайрфрейм-уровень, можно текстом+скетчем): главная с ответ-входом, словарь с двумя осями фильтров и кластерами, карточка термина (связи + якорь + слот визуала), индекс истории, страница главы с якорями.
3. **Модель таксономии и фильтров** — как Уровень × Профиль × Категория уживаются в UI.
4. **Паттерн визуала** — как встроены схема/видео.
5. **Claude Design-промпт(ы)** — готовые к вставке, по образцу `design_brief.md` (приложить лого, web capture текущего сайта, просить 2–3 направления, экспорт standalone HTML). Это твой основной артефакт для генерации макетов.
6. Заметка: какие изменения в данных это потребует (для дев-чата) — на уровне «нужны поля связей и якорей», без реализации.

PM примет результат, утвердит с Сашей, и только потом дев/контент-чат перестраивает данные и верстает — чтобы не делать дважды.

## 8. Решения по ходу (Q&A дизайнера, 14.06)

- **Формат сдачи:** док с IA + текст-вайрфреймы + Claude Design-промпт(ы). Отрисованные мокапы — не в Cowork, а в Claude Design по промпту.
- **Поле «спроси» (ответ-вход):** гибрид — Fuse.js как дефолтный роутер по терминам/ключам + курируемая карта «Вопрос→якорь» поверх для intent-вопросов («как поставить зонную оборону», «как стать скаутом»). Нет курируемого совпадения → результаты Fuse. Без LLM.
- **Идентичность:** не фиксируем заранее. Базлайн — эволюция в рамках ДНК «программки матча» (стиль свежий, завязан на узнаваемость канала; тяжёлая работа спринта структурная). Дать 2–3 направления + обязательно один смелый вариант с заметным сдвигом для сравнения. Финал — по рендерам.
