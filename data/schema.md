# Схема данных — hockey-platform

> Источник правды о структуре `terms.json`, `expert_review.json`, `clusters.json`,
> `history.json`, `quick_answers.json`.
> При изменении схемы — обновить `CLAUDE.md` тоже.
> Файлы дублируются: `data/` (источник правды) ↔ `docs/data/` (отдаётся сайтом). Держать в синхроне.

---

## terms.json

Массив объектов. Каждый объект — один хоккейный термин.

| Поле | Тип | Обязательное | Описание |
|------|-----|:---:|---------|
| `id` | string | ✓ | Уникальный slug (латиница, строчные, дефисы). Пример: `expected-goals` |
| `en` | string | ✓ | Каноническое английское название |
| `ru` | string | ✓ | Русский перевод или транслитерация |
| `category` | string | ✓ | `stat` / `tactic` / `position` / `rule` |
| `abbr` | string[] | ✓ | Массив сокращений. Пустой `[]` если нет |
| `definition` | string | ✓ | Объяснение по-человечески. Без академизма. 2–5 предложений |
| `ru_slang` | string\|null | ✓ | Разговорный русский аналог. `null` если нет |
| `profiles` | string[] | ✓ | Ось **Профиль** (мультизначение): `general/player/coach/analyst/scout`. `[]` пока не размечено — заполняет контент-чат по рубрикатору |
| `cluster` | string\|null | ✓ | slug родителя-кластера из `clusters.json`. `null` если термин вне кластеров |
| `see_also` | string[] | ✓ | id смежных терминов («см. также»). Горизонтальные связи, не иерархия |
| `anchor` | object\|null | ✓ | Якорь «подробнее» в канон-материал. `null` или `{ target, anchor_id, label }` (см. ниже) |
| `visual` | string\|null | — | Источник визуала. Схема — путь к SVG; видео — `youtube:VIDEO_ID`. `null` пока нет |
| `visual_type` | string\|null | — | `scheme` / `video` / `gif` / `null` |
| `visual_timecode` | string\|null | — | Только `video`: таймкод-якорь, напр. `"2:14"` |
| `visual_caption` | string\|null | — | Подпись к визуалу |
| `visual_attribution` | string\|null | — | Источник/автор. **Обязателен для `video`** |
| `diagram` | string | — | Легаси-путь к SVG-схеме (читает текущий `docs/app.js`). Для новых схем дублировать в `visual`; консолидация — отдельной задачей |
| `source` | string | ✓ | Источник или сайт для проверки |
| `status` | string | ✓ | `verified` — проверено. `unverified` — требует проверки |
| `added` | string | ✓ | Дата добавления ISO: `2026-06-07` |
| `notes` | string | — | Служебные заметки. Не отображаются в UI |
| `level` | string | ✓ | Ось **Уровень**: ключи `novice` / `fan` / `geek` (см. карту меток) |

### Ось Уровень — ключи vs метки в UI

Ключи в данных **не меняем** (`novice/fan/geek` уже у всех терминов). В интерфейсе — карта меток:

| Ключ (данные) | Метка (UI) |
|---|---|
| `novice` | База |
| `fan` | Средний |
| `geek` | Продвинутый |

Переименование ключей на `base/mid/adv` — опционально, отдельной задачей (это правка данных).

### Объект `anchor` (связь термина с каноном)

```json
"anchor": { "target": "history/origins.html", "anchor_id": "stanley-cup", "label": "История — гл. 1 «Истоки»" }
```
`target` — относительный URL страницы; `anchor_id` — id раздела (из `history.json → sections`); `label` — текст ссылки.

### Контракт визуала (схема / видео)

Один слот, два типа (`visual_type`):
- **scheme** — своя SVG. `visual` = путь (`diagrams/forecheck.svg`), стиль из `docs/diagrams/` держим.
- **video** — встраиваемое. `visual` = `youtube:VIDEO_ID`, `visual_timecode` = якорь, `visual_attribution` обязателен. Паттерн «фасад»: превью + ▶, iframe по клику.

### Категории

- **stat** — статистические метрики (Corsi, xG, PDO, Sv%)
- **tactic** — тактические концепции (форчек, бэкчек, трап, слот)
- **position** — позиции и роли (winger, center, stay-at-home D)
- **rule** — правила игры (вбрасывание, офсайд, icing)

### Правила заполнения

1. `id` уникален в пределах файла
2. `definition` — без Wikiped-стиля. Пишем как объясняем другу
3. Если термин не уверен на 100% — в `expert_review.json`, не сюда
4. `profiles` размечает контент-чат; дев только заводит поле
5. `cluster`/`see_also` — структурные связи, ведёт дев; целостность проверяет `tools/validate_data.py`

## clusters.json — реестр кластеров

**Решение: отдельный реестр, а не «термины-контейнеры».** Кластер («Броски», «Большинство») —
это группа, а не термин: у него нет определения/уровня/категории-сущности. Делать его записью
в `terms.json` — засорять словарь не-терминами и ломать поиск/фильтры. Поэтому кластеры живут
отдельным реестром, а термины ссылаются на него полем `cluster`. Если кластер совпадает с
существующим термином (напр. «Большинство» = `power-play`) — это отражается полем `term_id`.

| Поле | Тип | Описание |
|------|-----|---------|
| `slug` | string | Уникальный идентификатор кластера (на него ссылается `terms.cluster`) |
| `title` | string | Отображаемое имя группы («Броски») |
| `category` | string\|null | Категория группы (для подсказки в UI) |
| `term_id` | string\|null | id термина-родителя, если кластер = существующий термин (иначе `null`) |
| `order` | number | Порядок вывода |
| `summary` | string | Короткое описание группы |

Плоский A→Z-вид строится из терминов с `cluster: null` + всех остальных без группировки.

## history.json

Массив глав. Поля:

| Поле | Тип | Описание |
|------|-----|---------|
| `id` | string | slug главы (совпадает с именем файла `history/<id>.html`) |
| `title` | string | Заголовок главы |
| `summary` | string | Краткое описание (тизер) |
| `url` | string | Ссылка на страницу главы |
| `keywords` | string[] | Триггеры для бота/поиска (совпадение по границам слов) |
| `sections` | object[] | Якоря разделов главы: `[{ id, title }]` — цели deep-link от ответ-входа и бота |
| `published` | bool | `true` — опубликована; `false` — карточка «скоро» в индексе |
| `added` | string | Дата ISO |

`sections[].id` обязан совпадать с `id` соответствующего `<h2>` на странице главы.

## quick_answers.json — курируемые быстрые ответы (гибрид-движок)

Топ-вопросы для ответ-входа на главной: рендерятся плашкой «⚡ быстрый ответ» над Fuse-выдачей.
Ведётся вручную, пополняется из `bot/missing_queries.log`. Без LLM — маршрутизатор по якорям.

| Поле | Тип | Описание |
|------|-----|---------|
| `q` | string | Формулировка вопроса (как спрашивают) |
| `summary` | string | 1–2 строки ответа |
| `target_url` | string | `dictionary.html` или `history/<chapter>.html` |
| `anchor_id` | string | id термина (для `dictionary.html`) или id раздела главы (для истории) |

Целостность `anchor_id` проверяется `tools/validate_data.py`.

## expert_review.json

Та же схема, что `terms.json`. Разница:
- `status` всегда `"unverified"`
- `notes` содержит причину неопределённости и что нужно уточнить
- В UI отображаются с плашкой «⚠ требует проверки»

### Как перевести термин из expert_review в terms

1. Получить подтверждение от эксперта (тренер, аналитик, профессиональный источник)
2. Заполнить поле `source` с указанием реального источника
3. Изменить `status` на `"verified"`
4. Переместить объект из `expert_review.json` в `terms.json`
5. Закоммитить: `data: verified термин fisherman (source: тренер Иванов)`
